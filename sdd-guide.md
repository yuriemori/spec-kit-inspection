# Spec-driven development with AI

元記事:
https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit/

## 概要
この記事は、AIコーディングエージェントを使った開発で起こりがちな「それっぽいが意図とずれる実装」を減らすために、Spec Kit というオープンソースのツールキットを使った Spec-driven development（仕様駆動開発）を紹介しています。

主張の中心は、
- コードを先に書くのではなく、まず仕様を明確化する
- 仕様を「静的な文書」ではなく「実行可能で更新され続ける成果物」として扱う
- AIには曖昧な指示ではなく、段階的で検証可能な指示を与える

という点です。

## なぜ必要か
記事では、いわゆる「vibe-coding（雰囲気でプロンプトしてコードを出す）」は、
- プロトタイプには速い
- しかし本番品質や既存コードベースへの統合では不安定

と述べています。問題はAIのコーディング能力そのものより、開発者側の指示の与え方にある、という立場です。

## Spec Kit の4フェーズ
Spec Kit では、次の4段階を順に進めます。各段階で検証し、問題があれば次に進む前に修正します。

1. Specify
- 何を作るか、なぜ作るかを高レベルで定義
- ユーザー体験、成功条件、利用シナリオを明確化
- 仕様は学習に応じて更新される「生きた成果物」

2. Plan
- 技術スタック、アーキテクチャ、制約（既存システム連携、性能、コンプライアンスなど）を反映
- 必要なら複数案を比較
- 組織の標準や内部ドキュメントを計画に取り込む

3. Tasks
- 仕様と計画を、実装・レビューしやすい小さなタスクへ分解
- 「認証を作る」のような大タスクではなく、検証可能な粒度に落とす

4. Implement
- AIエージェントがタスク単位で実装
- 開発者は巨大な差分ではなく、焦点化された変更をレビュー

## 開発者の役割
AIに任せるだけではなく、各フェーズで人間が
- 妥当性を確認
- 欠落やエッジケースを発見
- 方向修正

することが重要だと強調されています。AIは成果物を生成し、人間は正しさを保証する、という役割分担です。

## 実際の使い方（記事内の流れ）
- specify コマンドで初期化
- /specify で仕様生成
- /plan で技術計画生成
- /tasks で実行可能タスクへ分解

この流れにより、曖昧な要求を、AIが実行しやすい明確な意図へ変換できます。

## この手法が効く理由
記事は「LLMはパターン補完は得意だが、意図の読心はできない」と説明します。
そのため、
- 仕様で「何を作るか」を固定
- 計画で「どう作るか」を定義
- タスクで「何から作るか」を順序化

することで、推測に頼る割合を減らせます。

また、セキュリティ方針・コンプライアンス・デザインシステム・連携要件など、組織知を仕様と計画に明示的に埋め込める点も利点です。

## 特に有効な3つの場面
1. 新規開発（0→1）
- 早期に仕様と計画を固めることで、汎用的すぎる実装を回避

2. 既存システムへの機能追加（N→N+1）
- 既存制約との整合を事前に明示でき、安全に拡張しやすい

3. レガシー刷新
- 失われた意図を仕様として再定義し、新アーキテクチャで再構築しやすい

## 方向性と今後
記事は「コードが唯一の真実」から「意図が真実」へ移行していくと述べます。AI時代には、仕様がそのまま実装に変換されるため、仕様品質が成果品質を左右する、という考え方です。

Spec Kit はその実践のためのオープンソース実験であり、今後は
- VS Code との統合
- 複数実装の比較・差分
- 大規模組織での仕様/タスク管理

といった改善領域に取り組む方針が示されています。

## 一言でまとめると
Spec-driven development は、AIに「うまく作って」と頼む開発から、仕様・計画・タスクで意図を構造化して「確実に作らせる」開発へ移行するための実践フレームワークです。
